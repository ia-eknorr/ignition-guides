# Contributing

## Information architecture

Content is organized **by theme**, not by reference architecture. The Guides
section holds concept-and-task pillars (Docker, Kubernetes, Observability,
Version Control); Labs holds end-to-end tutorials; Reference holds lookup
material. Each pillar follows the Kubernetes-docs Concepts / Tasks / Tutorials /
Reference convention. One-screen how-to pages are collected under a **Tasks**
sub-section inside their pillar once the pillar grows, rather than living in a
separate catalog.

Every category with children gets a landing page wired through
`link: { type: "doc", id: "<section>/index" }` (or an explicit `intro` doc) in
`sidebars.ts`. The sidebar is maintained by hand; it is not autogenerated.

## The `applies_to` front-matter convention

Pages that describe a deployment pattern declare which contexts they apply to
with an `applies_to` array in front matter:

```yaml
---
applies_to: [docker, kubernetes, bare-metal]
---
```

Valid values are `docker`, `kubernetes`, and `bare-metal`. The array renders as
a row of **Applies to** badges at the top of the page, so readers can tell at a
glance whether a page is relevant to their setup.

Rendering is automatic: the [`AppliesTo`](src/components/AppliesTo/index.tsx)
component is injected on every doc page through a swizzled
[`DocItem/Content`](src/theme/DocItem/Content/index.tsx) wrapper, which reads
`applies_to` from the page's front matter. Pages that omit the field render no
badge, so there is nothing to add per page beyond the front matter itself.

Omit `applies_to` on pages where deployment context does not apply (reference
tables, version-control guides, and similar).

## Linking discipline

- Cross-link to **pages**, not `#section` anchors, where practical. Anchors are
  validated only as warnings, so a broken anchor can slip past the build.
- Treat core-product concepts as prerequisite links to
  [docs.ia.io](https://docs.ia.io) and chart value semantics as links to
  [charts.ia.io](https://charts.ia.io). Do not re-explain or duplicate them.

## Build gate

`npm run build` must exit 0 **and** print no broken-link or broken-anchor
warnings. The build does not fail on broken Markdown links or anchors by
default, so scan the build output, not just the exit code.
